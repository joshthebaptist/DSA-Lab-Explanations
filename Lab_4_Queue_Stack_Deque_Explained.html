Core Concepts You Need to Know

Stack (LIFO)

Last-In, First-Out — like a stack of plates. The last plate you put on top is the first you take off.

push(x) → add to top
pop() → remove from top
peek() → look at top

Queue (FIFO)

First-In, First-Out — like a line at a cashier. The first person in line gets served first.

offer(x) → add to back
poll() → remove from front
peek() → look at front

Deque (Double-Ended)

Can add/remove from both ends. Works as both stack AND queue.

addFirst/addLast
removeFirst/removeLast

PriorityQueue

Elements are ordered by priority (not insertion order). Highest priority comes out first.

Uses compareTo() to decide order.

Key Insight: In Java, ArrayDeque is a class that implements both Queue and Deque interfaces. When you store it in a Queue variable, you can only call Queue methods (offer, poll, peek). When stored as Deque, you can also call push/pop (stack operations).

1. Palindrome Phrase Checker

What it does: Reads multiple test cases, checks if each phrase is a palindrome (reads same forwards and backwards, ignoring case and non-letters). Uses a stack and a queue together.

How the two files work together

PalindromeApp.java → The driver (main) that reads input and runs the checker.

PalindromeChecker.java → The logic class that does the actual palindrome checking using a stack and a queue.

⇓

PalindromeApp creates a Scanner, reads T test cases. For each test case, it creates a new PalindromeChecker, processes the input, then asks "is it a palindrome?" and prints Yes/No.

3. Patient Queue System (PriorityQueue)

What it does: Reads patient names and priority levels, then serves them in priority order (highest priority = lowest number first). Unlike a regular queue (FIFO), PriorityQueue serves based on importance, not arrival order.

How the three files work together

PatientQueue.java
Wrapper around PriorityQueue — provides addPatient and serveAllPatients methods.
PatientApp.java
Main program — reads input, creates Patient objects via the queue, then serves them.

Flow: PatientApp reads data → calls PatientQueue.addPatient() which creates a new Patient → PriorityQueue orders Patients by compareTo() → serveAllPatients() polls in priority order

★ Behind the Scenes — Step-by-Step Execution Trace

Let's trace what happens with input: n = 3, patients = Alice 1, Bob 3, Charlie 2.

Phase 1 — Inserting patients into the PriorityQueue

PriorityQueue uses a min-heap internally. The heap is a binary tree where the smallest element (by compareTo) is always at the root. After each offer(), the heap rearranges itself to maintain this property.

        /
   [Bob(3)]
Bob (priority 3) is added as a child of Alice. Since 3 > 1, no swap needed. Alice remains the root (highest priority).

Phase 2 — Serving patients (polling in priority order)

The serveAllPatients() method loops, calling pq.poll() each time. poll() removes the root of the heap (highest priority), then restructures the heap so the next smallest element rises to the root.

1st Alice (Priority: 1) Alice (Priority: 1) Alice has the highest priority (lowest number = 1), so she's served first.
2nd Bob (Priority: 3) Charlie (Priority: 2) Even though Bob was added before Charlie, Charlie has better priority (2 < 3), so Charlie jumps ahead.

Critical insight: PriorityQueue does NOT follow FIFO! Bob was added before Charlie, but Charlie was served before Bob. The service order is determined entirely by compareTo() — lower priority number = served first. This is why the Comparable interface matters: it tells the PriorityQueue how to determine which element is "smaller" (higher priority).
